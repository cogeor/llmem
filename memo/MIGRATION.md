# Migration Plan

How to get from today's `src/` tree to the structure proposed in `ARCHITECTURE.md` without breaking the build at any commit.

## Ground rules

- Every step ends with a green build and a green `npm test`.
- No "rename + edit" in one commit. Renames first, behavior changes after.
- Use `git mv` (not delete+add) so blame survives.
- One step = one PR. The plan is incremental on purpose.

## Step 0 — Cruft removal (no code moves)

Pure deletions and gitignores. Should be one PR.

- Delete: `design.txt`, `memo.txt`, `test-ws.js`, `llmem-0.1.0.vsix`.
- Delete: every committed `.d.ts` adjacent to a `.ts` in `src/` (regenerated by `tsc`).
- Add to `.gitignore`: `*.vsix`, `**/*.d.ts` under `src/`.
- Move `scripts/install-hooks.js` into `tools/`.

## Step 1 — Workspace setup

Convert the repo to npm workspaces. No file moves yet.

- Add `"workspaces": ["packages/*", "apps/*"]` to root `package.json`.
- Create empty `packages/` and `apps/` directories with placeholder `package.json` files.
- Build still runs the existing tsconfigs.

This step is a no-op at runtime but unlocks per-package builds.

## Step 2 — Extract `core`

Lowest-risk move because nothing depends on it being elsewhere.

- Create `packages/core/`.
- Move into it:
  - `src/common/logger.ts` → `packages/core/src/logger.ts`
  - `src/common/index.ts` → `packages/core/src/index.ts`
  - introduce `packages/core/src/types.ts` (workspace-root brand, path brands, error classes)
- Update imports across `src/` to `@llmem/core`.
- `tsconfig` for `packages/core` extends the new `tsconfig.node.json`.

## Step 3 — Extract `domain`

- Create `packages/domain/`.
- Move into it:
  - `src/graph/types.ts`
  - `src/graph/utils.ts` (pure helpers only — anything touching disk stays behind temporarily)
  - new `packages/domain/src/folder-tree.ts` (placeholder, populated by design/02)
- Update imports.

`packages/domain` must compile with **only** `core` imports. Add the architecture test for that direction at the end of this step (it will pass trivially today; it guards future drift).

## Step 4 — Extract `graph-store`

- Create `packages/graph-store/`.
- Move into it: `src/graph/edgelist.ts`, `src/graph/edgelist.test.ts`.
- Anything in `src/graph/index.ts` that builds graphs *from* edges moves to `packages/domain`. Anything that *persists* edges stays in `graph-store`.

## Step 5 — Extract `parsers/*`

- Create `packages/parsers/shared/` for the adapter interface (`src/parser/adapter.ts`, `interfaces.ts`, `registry.ts`, `config.ts`, `line-counter.ts`).
- Create `packages/parsers/typescript/` and move `src/parser/typescript/*`, `src/parser/ts-extractor.ts`, `src/parser/ts-service.ts`.
- Create `packages/parsers/{python,cpp,rust,r}/` and move each language folder accordingly.
- Each parser package declares its `tree-sitter-*` peer dep in its own `package.json`.

After this step, `peerDependencies` move out of the root `package.json` into per-parser packages — a cleaner story for users who only need one language.

## Step 6 — Extract `platform-fs`

This is the load-bearing step. Until here, workspace-root resolution is duplicated; after here, it has one owner.

- Create `packages/platform-fs/src/workspace-root.ts` consolidating the four current resolvers.
- Move `src/claude/server/file-watcher.ts` and `arch-watcher.ts` here.
- Update **every** call site to take a `WorkspaceRoot` (branded type) instead of a `string`. This is the change that will surface most hidden bugs — expect to write a few targeted tests as you go.
- The README's "Known Issue" about `report_file_info` saving to the wrong path is fixed by this step. Add a regression test.

## Step 7 — Extract `docs`

- Create `packages/docs/`.
- Move into it: `src/artifact/path-mapper.ts`, `src/artifact/storage.ts`, `src/artifact/tree.ts`.
- Delete `src/artifact/index.ts` (deprecated per `CLAUDE.md`). If anything still imports it, those callers move to `packages/docs/index.ts` API.
- The spec→file index from `design/03_spec_to_code_mapping.md` lands here when that work starts; leave a stub file for now.

## Step 8 — Extract `application`

- Create `packages/application/`.
- Move into it (renamed where useful):
  - `src/info/extractor.ts` → `packages/application/src/document/file.ts`
  - `src/info/folder.ts` → `packages/application/src/document/folder.ts`
  - `src/info/mcp.ts` → split: prompt-building stays in `application/document/`, MCP-specific glue moves to `platform-mcp` in step 9.
  - `src/info/renderer.ts` → `packages/application/src/document/render.ts`
  - `src/info/reverse-index.ts` → `packages/application/src/graph/reverse-index.ts`
  - `src/info/filter.ts` → `packages/application/src/graph/filter.ts`
  - the body of `src/scripts/scan_codebase.ts` and `generate_edgelist.ts` becomes `packages/application/src/scan/index.ts` (CLI entry stays in `apps/cli`)

After this step, `application` does not import any `platform-*` package or any `apps/*` entry. Add the boundary test.

## Step 9 — Extract `platform-mcp` and `platform-http`

- Create `packages/platform-mcp/`. Move `src/mcp/*`. Split `src/mcp/tools.ts` into one file per tool (`tools/file-info.ts`, `tools/folder-info.ts`, `tools/inspect-source.ts`, `tools/open-window.ts`, `tools/report-file-info.ts`, `tools/report-folder-info.ts`) plus `tools/register.ts`. Each tool file is < 200 lines.
- Create `packages/platform-http/`. Move `src/claude/server/http-handler.ts`, `websocket.ts`, `watch-manager.ts`, `index.ts`.
- Both packages depend on `application`, never the other way around.
- Delete the `import '../extension/config'` from the old `tools.ts` — `platform-mcp` reads config through `application`, not directly from the VS Code extension layer.

## Step 10 — Extract `viewer-shared` and split the webview

This is the deployment-artifact-integrity step.

- Create `packages/viewer-shared/`. Move `src/webview/ui/types.ts` and any other type-only modules.
- Create `apps/web-viewer/`. Move all of `src/webview/ui/**` here. Add `tsconfig.browser.json` extending the new browser base.
- Move `src/webview/generator.ts`, `design-docs.ts`, `data-service.ts`, `live-reload.ts`, `graph-status.ts`, `worktree.ts` into `packages/platform-http/src/page/` — they emit HTML on the server.
- Add `tests/arch/browser-purity.test.ts` — fails if `apps/web-viewer/**` imports any Node-only module.

## Step 11 — Promote the apps

- Create `apps/cli/`. Move `src/claude/cli.ts`, `bin/llmem` (now a generated wrapper).
- Create `apps/mcp-server/`. Move `src/claude/index.ts`, `src/mcp/server.ts`. The MCP server entry point reads from `apps/mcp-server/src/index.ts` and wires `platform-mcp` against `application`.
- Create `apps/http-server/`. Move `src/claude/server/index.ts`, `web-launcher.ts`. Composes `platform-http` + `application`.
- Create `apps/vscode-extension/`. Move `src/extension/*`. Composes `application` + `platform-mcp` (running in-process instead of stdio).
- Decide canonical packaging: `llmem-plugin/` becomes the **build output** of `apps/mcp-server`, generated under `dist/plugin/`. Delete the hand-maintained `llmem-plugin/dist/cli.bundle.js`.

## Step 12 — Test consolidation

- Move every `*.test.ts` colocated in `src/` to `tests/unit/<package>/<name>.test.ts`.
- Promote useful `verify_*.ts` and `test_*.ts` scripts to real tests under `tests/integration/`. Delete the rest.
- Add the architecture tests under `tests/arch/`:
  - `dependencies.test.ts` — parses imports across the workspace, asserts the rules in `ARCHITECTURE.md`
  - `workspace-root.test.ts` — every `fs.write*` outside `platform-fs` is forbidden
  - `browser-purity.test.ts` — see step 10
  - `file-budget.test.ts` — enforces the line-count budgets

## Step 13 — `infra/` and `docs/`

- Create `infra/` for the platform work (see `design/04`). Empty for now; populated when the platform spec is implemented.
- Move user-facing docs: `README.md` stays at root, but `CHANGELOG.md`, `CONTRIBUTING.md`, and `docs/` consolidate.

## Verification at each step

After each step, the following must hold:

```
npm install
npm run build:all
npm test
npm run lint
```

If any of these fail, the PR is incomplete. Do not ship a "next step will fix it" PR — the architecture memo's last rule applies here too.

## Estimated sequencing

Steps 0–4 are mechanical and can land back-to-back over a week. Step 6 (workspace root unification) is the highest-risk step and should be its own PR with extra tests. Steps 7–10 are the real architectural shift and are where the team should slow down. Steps 11–13 are polish.

After step 11, the README and `CLAUDE.md` need a rewrite — the deprecated paragraphs about `src/artifact/` and the workspace-root known issue both go away.
