# Software Platform

Status: design sketch.
Owners: infra/, new private repo (control plane), apps/web-viewer (consumer of artifact bundles).

## Vision

A user pastes a git URL into a web form. The platform clones the repo, runs LLMem, and serves the resulting artifact. The user gets a shareable link to a browsable, queryable codebase view without installing anything.

This is the DigitalOcean App Platform shape: hosted control plane + buildpack-style indexing pipeline + static artifact storage.

## Two-tier licensing

This is the load-bearing constraint:

- **The viewer** (`apps/web-viewer`) and the entire local toolchain (everything in this repo today) stays **open source**. Users can run LLMem locally against their own repo for free, forever.
- **The platform** — the control plane that ingests git URLs, runs indexers, hosts artifacts, manages auth and billing — stays **private** and **closed source**. It lives in a separate repo (e.g. `llmem/platform`).

The artifact format is the contract between them. As long as the platform produces an artifact bundle that the open-source viewer can load, users have a meaningful exit path: download the bundle, host it themselves.

## Artifact bundle format

The contract between platform and viewer. **Versioned.** **Stable.** This is what the viewer reads, period. The bundle is a tarball or a folder under `.artifacts/` — the local viewer reads the latter; the hosted viewer reads the former.

```
bundle/
├── manifest.json              # version, repo URL, commit SHA, indexed-at, languages
├── import-edgelist.json       # same format as today
├── call-edgelist.json
├── folder-tree.json           # design/02
├── spec-index.json            # design/03
├── arch/                      # the .arch/ shadow filesystem from the indexed repo
│   └── ...
├── source-snapshot/           # optional: a sparse snapshot of the source for inspect_source
│   └── ...
└── README.md                  # human-readable summary, generated by the indexer
```

The viewer takes a `BundleSource`:

```ts
type BundleSource =
  | { kind: 'local'; root: AbsPath }                  // current behavior
  | { kind: 'http'; manifestUrl: string };            // hosted bundle
```

`apps/web-viewer` already has a `dataProvider` abstraction (`src/webview/ui/services/dataProvider.ts`); this is the formal seam.

## Privacy / security shape

The control plane is the only component that touches user credentials and source. The artifact bundle is **derived data** — it can be made public per-user-policy without exposing the source repo.

- Bundles default to **private**: only the submitter can view, via short-lived signed URLs.
- Public-share is opt-in per-bundle. The artifact contains derived structure, not full source unless the user explicitly opts into `source-snapshot/`.
- The job runner is sandboxed: each repo clones into a fresh container, runs LLMem + SCIP indexers (see `design/01`), uploads the bundle, container is destroyed.
- No third-party telemetry from the indexed code path. The marketing site's analytics are scoped to the marketing site.

## Job runner

Per indexed language, run:

1. `git clone --depth=N <repo>` (depth configurable, default 1; full clone for paid tiers).
2. `llmem scan --workspace . --output ./bundle` — produces edge lists + folder tree + .arch (currently empty unless an agent generates it).
3. `scip-{python,typescript,clang,rust} ...` per detected language → `.scip` files.
4. `llmem index --convert-scip ./bundle/scip --output ./bundle` — folds SCIP into call edges.
5. `llmem specs link --workspace .` → `bundle/spec-index.json`.
6. `tar czf bundle.tar.gz ./bundle && upload`.

The runner runs in containers. The control plane manages a pool. Per-repo billing is "minutes of indexer time + GB-months of artifact storage."

## Why two-tier instead of fully OSS

- Keeps a sustainable revenue surface. The viewer being free + great is the lead magnet; the platform pays the bills.
- Centralizes operational complexity (billing, auth, multi-tenant compute) where it belongs — out of the viewer codebase.
- Lets us iterate on indexer infrastructure (containers, queues, autoscaling) without polluting the OSS repo with deploy specs and ops scripts.

## Why the viewer must stay OSS

- Trust. Users are pasting git URLs of their own code. They need to know the rendering layer they're staring at is auditable.
- Self-hosting escape hatch. If the platform shuts down, users can still load their bundles in the viewer.
- The Claude integration story (design/05) is "drop a CLI on your machine, point at a remote bundle." That CLI cannot be closed-source — half its value is offline support.

## Architecture placement

This repo, today:

- `apps/web-viewer` already supports loading data via a provider abstraction. Add `HttpBundleProvider`.
- `apps/cli` gains `llmem pull <bundle-url>` to fetch a hosted artifact and serve it locally. Useful for offline review.
- `packages/viewer-shared` defines the manifest format.
- `infra/bundle-format/` (new) holds the JSON Schema and a `validate-bundle` test harness so platform PRs can confirm conformance.

The private platform repo:

- `apps/control-plane/` — FastAPI or Go service.
- `apps/marketing/` — landing page.
- `infra/runner-image/` — Docker image for the indexer worker.
- `infra/deploy/` — DO App Platform spec, Spaces buckets, etc. Apply rules 76–91 from aipr's `best_practices.md` here, especially: same-origin rewrite-proxy over host-based ingress, never bind `DATABASE_URL` to a managed component when the DB is external, secrets in `.env.prod`, render-on-deploy.

## Open questions

- Pricing shape: per-repo flat, per-indexer-minute, or tiered by language coverage? Out of scope here — design/04 only commits to "the thing has a billing surface."
- Embed mode: do we let users iframe the viewer for their own dashboards? Probably yes, but post-v1.
- Diff mode: comparing two bundles (last week vs today). Strong product fit; defer the spec until the artifact format is stable.
- Free tier: probably "small public repos, indexed once, expire after 30 days." Keeps the free path useful as a demo without becoming a free CI service.
