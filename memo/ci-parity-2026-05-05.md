# Closing the local-pass / CI-pass gap (2026-05-05)

## Why this memo exists

The 2026-05-05 codebase-quality stream landed 17 green commits locally on
Windows. The first time CI actually ran, three independent failures surfaced
in sequence ‚Äî not introduced by the stream, but latent for over a month:

1. `package.json`'s `node --test "tests/unit/**/*.test.ts"` fails on
   Node 20 + dash (Ubuntu CI's `/bin/sh`). `**` was passed as a literal,
   so `test:unit` exited 1 before reaching `test:integration`.
2. `pretest` did not run `build:claude`, and `build:claude` was itself
   broken since `05959a4` (2026-03-22) ‚Äî `tsconfig.claude.json`'s narrow
   `rootDir: src/claude` cannot emit cross-tree imports the CLI uses.
   So `tests/contracts/cli-schema.test.ts:ensureBuilt()` would have
   thrown even if (1) hadn't.
3. `tests/integration/arch-watcher.test.ts` listened for a WebSocket
   `arch:deleted` event with `ws.once('message')`. On Linux's inotify,
   the prior `arch:created` is queued in the WS by the time `once`
   registers; on Windows's ReadDirectoryChangesW it collapses, so the
   test passed locally and failed on CI.

Each bug masked the next ‚Äî fixing one only revealed another. The team-wide
question is not "what was the bug" but "how do we guarantee, as much as
possible, that local-pass implies remote-pass."

## Where local and CI actually diverge

| Axis | Local (Windows in this case) | CI (`ubuntu-latest`) |
|------|------------------------------|----------------------|
| OS file events | ReadDirectoryChangesW | inotify ‚Äî different timing, batching, ordering |
| Path semantics | Case-insensitive, `\` separator | Case-sensitive, `/` separator |
| Shell for npm scripts | PowerShell / cmd | dash ‚Äî no `**` globstar, no bashisms |
| Node version | Whatever you have | Matrix (currently 20.x + 22.x) |
| `dist/`, `node_modules/` state | Often stale from prior runs | Fresh `npm ci` every time |
| Optional native deps | What happened to install | Whatever `npm ci` resolves |
| Line endings | CRLF | LF |
| Locale, timezone, scheduler | Workstation defaults | Container defaults |

You cannot eliminate every axis. The goal is to make divergence rare
enough that "passes locally" is genuinely informative.

## Tiered mitigations

### Tier 1 ‚Äî Cheap, immediate

1. **Test from a clean state.** Add `npm run verify` =
   `rimraf dist && npm ci && npm run build:all && npm test`. Run it before
   pushing anything non-trivial. Would have caught bug 2 the day it landed.
2. **Make `package.json` scripts dash-compatible.** No `**/*` globs, no
   PowerShell-only constructs, no Windows-only flags. (Done ‚Äî see
   `scripts/run-tests.cjs`.)
3. **`.gitattributes` for line endings.** `* text=auto eol=lf` so the
   repo content is byte-identical between Windows checkouts and CI
   checkouts. Removes one whole class of "works on my machine."
4. **Branch protection on `main`** requiring CI green. Cannot accept a red
   baseline that nobody pushes against. *(Not changeable in code ‚Äî must be
   set in GitHub repo settings ‚Üí Branches ‚Üí Branch protection rules.)*

### Tier 2 ‚Äî The single most valuable change

5. **Run CI in Docker locally with [`act`](https://github.com/nektos/act).**
   `act` reads `.github/workflows/ci.yml` and executes the matrix in a
   container that mirrors the GitHub runner image. The closest thing to
   "CI passes on my machine ‚áí CI passes remote." Ship as
   `npm run ci:local`. Requires the user to have Docker and `act`
   installed; the script itself is one line.
   - Caveat: `act` is excellent for build/test parity but does not perfectly
     emulate every action. Treat it as ~95 % parity, not a substitute for
     the real CI run.

### Tier 3 ‚Äî Code-level resilience for the residual flake categories

6. **Treat filesystem-event tests as flaky by default.** Always filter by
   predicate (`waitForWsMessageOfType` instead of `once('message')`).
   Pin chokidar's `awaitWriteFinish`/`atomic` settings explicitly rather
   than relying on platform defaults.
7. **Cap concurrency for filesystem suites.** `--test-concurrency=1` for
   `tests/integration/` removes the inter-test event interleaving class
   of bugs ‚Äî slower, but deterministic.
8. **`path.posix` for any path that hits a snapshot or string compare.**
   Catches separator and case-sensitivity drift cheaply.
9. **One Linux dry-run before merge** if you develop on Windows. WSL2 or
   a 30-second `act` run covers nearly all of axis 1 + 2 + 5 above.

### Tier 4 ‚Äî Heaviest, only if the above isn't enough

10. **Devcontainer**: develop inside the same image CI uses. Eliminates
    the OS axis entirely. Highest cost, highest assurance. Worth it for
    teams; usually overkill for a solo project.

## Recommendation for this repo, in order of payoff

1. **Tier 2.5** ‚Äî `npm run ci:local` via `act`. One script, ~3 min run,
   catches all three of the bugs that bit us.
2. **Tier 1.1** ‚Äî `npm run verify` as the no-Docker fallback.
3. **Tier 1.3** ‚Äî `.gitattributes` for LF normalization.
4. **Tier 3.7** ‚Äî concurrency=1 for `tests/integration/`.
5. **Tier 1.4** ‚Äî branch protection on `main` (manual GitHub setting; not
   in this commit).

## Honest ceiling

Even with every mitigation above, timing-sensitive tests and
hardware-speed differences will occasionally diverge between a fast dev
laptop and a constrained CI runner. Two principles help:

- **Write tests that don't depend on timing where possible** (predicates,
  not sleeps; events, not polling).
- **When timing must matter, use generous, explicit windows** and assume
  CI is slower than your laptop, not faster.

The aim is "divergence is the exception worth investigating," not
"divergence is impossible."
